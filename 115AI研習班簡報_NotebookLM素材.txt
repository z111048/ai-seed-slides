────────────────────────────────────────────────────────────────
【第一頁】痛點現場 — 每案必審、每次必改
評分維度：問題清晰度
────────────────────────────────────────────────────────────────

■ 受影響對象 × 發生環節

受影響對象：7 名科室承辦人

業務流程：
  案件受理 → 初審逐頁核對 → 補正意見撰寫 → 複查版本差異 → 審議核定

痛點 A（初審逐頁核對）：
  逐頁確認章節完整性、書圖一致性、法規檢討、圖面標示等項目，規則明確但耗時。

痛點 B（複查版本差異）：
  每次補正後須比對前後版本差異；不熟悉規範的事務所需承辦人逐頁說明，大幅增加複查負擔。

■ 量化規模

  科室承辦員額：7 人
  年均受理案量：約 100 件 / 年
  當前程序中案件：59 件
  每次補正複查耗時：約半天（4 小時）/ 次
  平均補正輪次：2 至 4 次 / 案
  全科年耗初審複查：約 200 人日 / 年

■ 近 5 年審議淨工作天（實績數據）

  全員委員會：平均 70.4 天（36 件核定）
  專案委員會：平均 61.1 天（46 件核定）
  簡化委員會：平均 51.4 天（111 件核定）
  幹事會：平均 45.4 天（96 件核定）
  簽證案：平均 22.5 天（39 件核定）

  說明：初審與補正複查作業為可量化縮減的前段時間。

■ 不解決的代價

  - 200 人日 / 年流失於重複性格式比對
  - 人才離職後隱性知識歸零，無法傳承
  - 不熟悉規範的事務所拉長整體審議週期
  - 承辦人無暇聚焦設計品質的實質審查


────────────────────────────────────────────────────────────────
【第二頁】AI 解方 — 多技術整合，分階段落地
評分維度：解方可行性
────────────────────────────────────────────────────────────────

■ AI 技術地圖 × 應用場景

  技術一：大型語言模型（P1 優先）
    工具：訂閱制（Claude.ai / Microsoft Copilot）
    解決痛點：格式檢核、補正意見草稿生成、法規問答

  技術二：文件 OCR（P1 優先）
    工具：Azure Document Intelligence / pdfplumber
    解決痛點：報告書 PDF 轉為結構化章節與圖表

  技術三：影像辨識（P2 優先）
    工具：Vision LLM（訂閱 Multimodal）
    解決痛點：書圖一致性比對、圖面標示自動確認

  技術四：語音辨識（P2 優先）
    工具：Whisper / Azure Speech
    解決痛點：委員會 × 幹事會會議逐字稿自動產出

  技術五：知識庫 RAG（P3 優先）
    工具：本地向量資料庫
    解決痛點：法規條文比對、歷史補正意見檢索

■ 導入策略：先訂閱 → 再客製

  - 初期以訂閱制工具驗證需求，不綁 API 帳單，採購流程簡單
  - 機敏欄位去識別化後送入 AI Agent 輔助；雲端資源須待資安審查
    與系統穩定性評估通過後才導入，不在 POC 階段使用
  - POC 驗證有效後，視需求評估 API 或自建模型
  - 現有都設審議 Bot 平台作為整合基礎，擴充 AI 模組

■ 已盤點數據資產

  - 歷史審議案件：3,884 件（含報告書、進度、核定資料，已入庫）
  - 委員會 × 幹事會會議紀錄：2,882 場（近 15 年）
  - 決議 × 審查意見全文：97,127 條（已結構化入庫）
  - 常見補正意見大全（近 5 年）
  - 都市設計相關法規、審議原則

■ 預算概估（含持續費用）

  一次性費用：
    - POC 訂閱費：約 NT$9,000（3 個月）
    - 人力投入：承辦人力約 30 人日（內部消化）
    - 系統建置（OCR + LLM 模組開發 + 資安委外強化）：NT$100–200 萬

  每月持續費用：
    - LLM 訂閱費（Team 方案 × 7 人）：NT$7,000–12,000 / 月
    - OCR + 語音辨識 API（視案量）：NT$1,500–3,000 / 月
    - 雲端主機 / 向量 DB 儲存：NT$3,000–8,000 / 月
    - 系統維護費（承包商）：NT$8,000–20,000 / 月
    - 每月合計：NT$2–4 萬 / 月（年約 NT$25–50 萬）

  說明：
    - 雲端資源於資安審查通過後才導入，POC 期間以訂閱制為主
    - 資安委外採政府共同供應契約框架，免重新招標
    - 總擁有成本（TCO）= 一次性採購 + 年持續費用，
      申請城鄉發展基金時應涵蓋首年維運費用

■ 優先落地項目

  P1（立即可做）：
    - 報告書章節完整性自動檢核
    - 補正意見草稿生成
    - 會議前案件重點摘要

  P2（次階段）：
    - 會議逐字稿自動生成（語音辨識）
    - 版本差異 × 補正回應追蹤

  P3（後期推進）：
    - 書圖一致性影像辨識檢核

【第三頁】短期可驗證 — 90 天 POC 計畫
評分維度：解方可行性 × 可驗證
────────────────────────────────────────────────────────────────

■ 第 1 個月 — 資料基礎建設

  工作項目：
    - OCR 處理近 3 年報告書（約 300 件）
    - 結構化法規條文 × 補正意見庫
    - 法規組 × 歷史資料組完成 Skill 技能包
    - 建立向量索引（RAG 知識庫）
  里程碑：知識庫可查詢，回答準確率 ≥ 80%

■ 第 2 個月 — 內部 POC 開發 × 試用

  工作項目：
    - 訂閱制工具（Claude.ai / Whisper）串接報告書檢核
    - 補正意見草稿生成器初版上線
    - 科室承辦員試用 5 件真實案件
    - 收集人工核對 vs AI 初稿比對結果（50 件樣本對照）
  里程碑：格式檢核準確率 ≥ 85%，補正意見可用率 ≥ 70%
  基線說明：以人工初審結果為對照，50 件真實案件作為樣本

■ 第 3 個月 — Beta 測試 × 效益驗證

  工作項目：
    - 第一波：以機關內部及公家單位案件測試，確認系統穩定性
    - 第二波：邀請常送件五大事務所案件測試，收集補正意見清晰度回饋
    - 評估 Vision LLM 圖面辨識誤判率（列為獨立風險項）
    - 完成功能需求清單，安全性評估，規劃正式採購
    - 向長官呈報效益報告，研議資安審查時程與雲端導入可行性
  里程碑：試辦案件複查時間縮短 ≥ 30%（以人工複查時間為基線對照）

■ KPI 成效目標

  - 複查工時縮減 ≥ 30%（90 天驗證目標）
  - 格式檢核準確率 ≥ 85%（POC 第 2 個月）
  - 5 家事務所 Beta 測試完成（POC 第 3 個月）
  - 60+ 人日 / 年釋放（目標成效，保守估計）

■ 分階段分模組開發計畫書

  Phase 1（月 1–3）：
    模組 A — 文件 OCR 解析
      技術：Azure Doc Intelligence / pdfplumber
      里程碑：報告書可結構化查詢
    模組 B — RAG 知識庫建立
      技術：本地向量 DB + 法規 × 補正意見
      里程碑：問答準確率 ≥ 80%

  Phase 2（月 4–6）：
    模組 C — 格式 × 章節完整性檢核
      技術：LLM + Rule Engine（訂閱制）
      里程碑：格式準確率 ≥ 85%
    模組 D — 補正意見草稿生成
      技術：LLM + 歷史補正意見 RAG
      里程碑：可用率 ≥ 70%

  Phase 3（月 7–12）：
    模組 E — 語音辨識逐字稿
      技術：Whisper / Azure Speech
      里程碑：會議逐字稿自動產出
    模組 F — 版本差異 × 補正追蹤
      技術：Diff Engine + LLM 摘要
      里程碑：版本比對自動標記
    模組 G — 書圖一致性影像辨識
      技術：Vision LLM（待資安評估）
      里程碑：圖面誤判率 < 10%

■ 產品 POC — Claude Code 快速開發

  - 使用 Claude Code 開發 POC 雛型，可快速迭代驗證
  - 第一波測試：機關內部及公家單位案件
    （控制資安風險，驗證系統穩定性）
  - 第二波測試：常送件五大事務所案件封閉測試
    （驗證真實效益，收集使用回饋）
  - 穩定性與資安評估通過後，再由廠商協助正式化
